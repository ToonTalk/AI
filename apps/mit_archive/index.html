💡 Key Project Reincarnated: The Director Language

In the late 1970s, you designed and built Director—a pioneering Actor-based animation language. In Director, visual items on screen are "actors" that communicate by passing messages in a parallel scheduler. It allowed Logo-like turtle geometry, but with objects that could operate simultaneously and respond dynamically.

To try out a live, visual reincarnation of Director, open the Director Simulator tab from the sidebar. You can write commands or load pre-built scripts to watch multiple actors execute drawing procedures concurrently!

👑 ANI: The Cinderella Story Planner (1979)

Built directly on top of Director, ANI is an early AI story generator. It translates abstract character descriptions (like ugly stepmother or good fairy_godmother) and interpersonal relationship rules into parallel animation sequences.

Instead of manually keyframing movements, researchers write high-level story constraints. ANI's MacLisp solver schedules the subscenes and compiles them into concurrent message-passing commands for Director actors.

📐 DIAGRAMER: Interactive Constraint Solver

DIAGRAMER (implemented in diagrm.92) is a MacLisp layout solver for drawing structured box-and-arrow diagrams (up to 4 nodes) without overlaps and with minimized link crossings.

If box relations cause lines to cross, DIAGRAMER executes localized "expert" heuristics to untangle them—swapping nodes on the bottom row if the layout is horizontal, or on the right row if vertical.

📟 Marvin Minsky's Turtle Terminal (TT2500)

The Turtle Terminal (TT2500) was a custom vector graphics terminal designed by Marvin Minsky at the MIT AI Lab in the early 1970s. Rather than using raster scanlines, it drew using continuous vector strokes on a dedicated CRT display. Named for its target retail price of $2,500, the terminal featured a 16-bit processor built from discrete TTL logic, though limited control store memory required it to run as a remote satellite display terminal linked to host computers like the PDP-11 or PDP-10.

What makes the TT2500 ontologically distinct from standard Logo is its display model: the terminal does not deposit static ink trails. Instead, Danny Hillis wrote microcode that refreshes the CRT screen dozens of times per second by interpreting a **dynamic display list**. Commands like FD and RT deposit static geometry into this list. In contrast, SPIN (angular rotation velocity) and MOVE (translation velocity) define *velocities* that the hardware refresh loop continuously increments on every pass.

Because motion composes through nested coordinate frames, recursive procedures automatically stack these frames like a spinning plate mounted on another spinning plate. This allows extremely simple recursive shapes to turn into complex, articulated kinematic mechanisms—causing segments to dynamically counter-rotate, slide, and unfurl on screen.

You can read the original drawing procedures you wrote in 1979 in the Lisp Source listing, or load them directly into our vector CRT simulator to experiment with these dynamic, nested coordinate frames!

📖 Director Message Cheat Sheet

Create / Select Actor: (ask <name> <message>)

Sends a message to an actor. If it doesn't exist, it creates a new turtle actor at the center.

Parallelism & Ordering: (sequentially <msg1> <msg2> ...) (simultaneously <msg1> <msg2> ...)

📖 Kinematic display lists

The TT2500 did not draw static ink trails. Its microcode, written by Danny Hillis, interpreted a dynamic display list refreshed dozens of times a second.

FD and RT deposit static shape geometry. SPIN and MOVE carry background velocities (angular and translational) that the refresh loop continuously increments.

Recursion automatically stacks these dynamic coordinate frames (like a spinning plate mounted on another spinning plate), producing jointed kinematic mechanisms with articulated epicyclic motion.

Refresh Rate: To maintain a flicker-free image, the original vector CRT refreshed its display list dozens of times per second (typically 30Hz–60Hz). An Emulation Speed of 1.0 steps time t by 1 per frame, representing a real-time ~60Hz update rate.
